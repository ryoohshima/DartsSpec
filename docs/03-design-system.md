# 03. デザインシステム / UI・UX

対応ステップ: **Step 3 — デザインシステムと UI/UX の設計**

既存のダーツ関連サイト（ガチャガチャした EC サイト風）にはない「モダンさ」が最大の武器になる。**ここの設計品質がサービスの差別化の core** であるため、徹底して作り込む。

## 1. ビジュアルの方向性

**ダークモードを基調とした、サイバーで洗練されたデザイン**を第一候補とする。金属バレルの質感・ダーツバーの雰囲気と親和性が高く、数値やスペックが映える。

### デザイン原則

1. **Less is More** — 情報を詰め込まず、余白でスペックを際立たせる。
2. **数値が主役** — 総重量・全長といった数字を、大きく・美しく見せる（プレイヤーが最も気にする情報）。
3. **静的でなく動的** — パーツ選択に対して数値がアニメーションで反応し、「触って楽しい」を演出。
4. **シェア前提** — あらゆる画面が「スクショしたくなる / 貼りたくなる」美しさを持つ。

## 2. カラートークン（ダークテーマ・確定）

以下を正式なカラートークンとして**確定**する（#14）。実装では Tailwind theme / CSS 変数としてこの値を一元管理する。

| トークン | 用途 | 確定値 |
|---|---|---|
| `--bg-base` | 背景（最暗） | `#0A0A0F` |
| `--bg-surface` | カード面 | `#14141C` |
| `--bg-elevated` | 浮いた要素 | `#1E1E2A` |
| `--text-primary` | 主要テキスト | `#F5F5F7` |
| `--text-secondary` | 補助テキスト | `#9A9AAF` |
| `--border` | 境界線 | `#2A2A38` |
| `--accent` | アクセント（CTA・強調） | `#00E5C7`（サイバー系ティール） |
| `--accent-alt` | 第二アクセント | `#7B5CFF`（バイオレット） |
| `--danger` | エラー | `#FF5C7A` |

> アクセントは「ネオン」に寄せると安っぽくなりやすい。彩度を抑えた**上品なネオン**を狙う。ライトテーマは初期スコープ外とし、まずダーク一本に集中する。

### 2.1 コントラスト検証（WCAG AA）

主要な組み合わせのコントラスト比を算出し、AA 基準（通常テキスト 4.5:1 / 大きい文字・UI 部品 3:1）への適合を確認済み。

| 前景 / 背景 | 比率 | 判定 |
|---|---|---|
| `--text-primary` / `--bg-base`〜`--bg-elevated` | 15.13〜18.14 | AA ✓（AAA も満たす） |
| `--text-secondary` / `--bg-base`〜`--bg-elevated` | 5.98〜7.16 | AA ✓ |
| `--accent` / `--bg-base`・`--bg-surface` | 11.37〜12.26 | AA ✓ |
| `--bg-base`（文字）/ `--accent`（CTA ボタン） | 12.26 | AA ✓ |
| `--danger` / `--bg-surface` | 6.16 | AA ✓ |
| `--accent-alt` / `--bg-surface` | 4.20 | 大きい文字・ノンテキストのみ ✓ |

- **`--accent-alt` の使用制限**: 通常サイズの本文テキスト色としては使わない。大きい文字（18pt / 太字 14pt 以上）・境界線・装飾・グラデーションに限定する。

## 3. タイポグラフィ（確定）

以下を正式なフォント構成として**確定**する（#15）。

| 用途 | 確定フォント | 補足 |
|---|---|---|
| 数値（総重量・全長） | **Inter**（`font-feature-settings: "tnum"`）・weight 700 | 桁が揃う等幅数字で「計器」感を出す |
| 見出し | **Inter** + **Noto Sans JP**（フォールバック） | 欧文は Inter、和文は Noto Sans JP |
| 本文 | **Noto Sans JP** | 可読性重視 |
| OGP 画像内 | **Noto Sans JP**（サブセット化して埋め込み） | [§6.4](#64-ogp-実装上の注意) 参照 |

- 数字は **tabular figures（等幅数字）** を使い、アニメーション時にガタつかないようにする。
- Web フォントは Google Fonts（`Inter` / `Noto Sans JP`）を利用し、`font-display: swap` で FOIT を避ける。

## 4. 主要画面と UI コンポーネント

### 4.1 画面一覧

| 画面 | 役割 | 認証 |
|---|---|---|
| ランディング（LP） | サービス紹介・登録導線 | 不要 |
| サインアップ / ログイン | 認証 | 不要 |
| セッティング作成 / 編集 | パーツ選択・リアルタイム合算 | 必要 |
| マイページ（一覧） | 自分のセッティング一覧 | 必要 |
| 公開セッティングページ | シェア先が見る画面・OGP 対象 | 不要（閲覧） |
| 作成完了 / シェア画面 | シェアボタンを大きく配置 | 必要 |

### 4.2 中核コンポーネント: セッティングカード

本サービスの顔となる「カード型 UI」。テキストと数値ベースで、1 枚で 1 セッティングを美しく表現する。

```
┌─────────────────────────────────────┐
│  MY SETTING                          │
│  「〇〇スペシャル」        by @user   │
│                                      │
│   TOTAL WEIGHT      TOTAL LENGTH     │
│     20.5 g            65.0 mm        │  ← 数値が主役・大きく
│                                      │
│  ─────────────────────────────────  │
│  BARREL   TRiNiDAD  Gomez Type 12    │
│  SHAFT    L-style   190 Silent       │
│  FLIGHT   Fit Flight Standard        │
│  TIP      COSMO     Fit Point PLUS   │
│                                      │
│              [ X でシェア ]           │
└─────────────────────────────────────┘
```

- パーツ行はカテゴリラベル + ブランド + 製品名で構成（アイコンは初期リリースでは使わず、テキストベースで統一）。
- 総重量・全長は最上部に大きく。プレイヤーが一番見たい情報を最優先配置。

#### レイアウト仕様（確定・#15）

| 項目 | 確定値 |
|---|---|
| カード背景 / 枠 | `--bg-surface` / `--border` 1px・角丸 16px |
| カード内余白 | モバイル 20px / デスクトップ 32px |
| セッティング名 | Noto Sans JP・20〜24px・`--text-primary` |
| 作者（by @user） | 14px・`--text-secondary` |
| 数値（総重量・全長） | Inter 700・40〜56px・tabular figures・`--text-primary`、単位（g / mm）は 0.4em・`--text-secondary` |
| 数値ラベル（TOTAL WEIGHT 等） | Inter 600・11px・letter-spacing 0.1em・大文字・`--text-secondary` |
| パーツ行 | カテゴリラベル（Inter 600・11px・大文字・`--accent`）+ ブランド（`--text-secondary`）+ 製品名（`--text-primary`）、行間は 12px、区切りは `--border` の水平線 |
| 最小幅 | **280px** から崩れないこと。数値 2 カラムは 320px 未満で縦積みに切替 |

- 未選択のパーツ行は「未選択」と `--text-secondary` で表示（行自体は消さず、構成の全体像を保つ）。
- 一部パーツのスペックが未登録の場合、数値の近くに「参考値」の注記を添える（[02 §4](./02-parts-data.md)）。

### 4.3 インタラクティブ合算表示

- パーツをセレクトすると、総重量・全長の数字が**カウントアップ / カウントダウンのアニメーション**で変動する。
- 実装は数値補間（例: framer-motion の `useSpring` / `animate` や `react-countup` 相当）。
- 「触って数字が動く」体験そのものが、シェアの動機になる。

## 5. 「シェアしたくなる」動線設計

CGM マーケティングの生命線。シェアの摩擦を極限まで下げる。

### 配置（確定・#17）

| 画面 | 配置 |
|---|---|
| 作成完了（保存直後） | セッティングカード直下に「X でシェア」ボタンを**最大サイズ**（幅いっぱい・`--accent` 塗り）で配置。その下に「URL をコピー」を併設 |
| マイページ（一覧） | 各セッティングカードに「シェア」「URL コピー」のアイコン付きボタンを常設 |
| 公開ページ `/s/{id}` | カード下部に「X でシェア」「URL をコピー」を配置（閲覧者の再シェアを促す） |

- シェアボタンは Web Intent（`https://twitter.com/intent/tweet?...`）で、定型文 + ハッシュタグ + 公開 URL を差し込む。
- 「URL をコピー」は `navigator.clipboard` を用い、成功時に「コピーしました」のフィードバックを 2 秒表示する。

### 定型文（確定・#17）

```
自分のダーツセッティングを晒してみた🎯
「{セッティング名}」
総重量 {total_weight_g}g / 全長 {total_length_mm}mm

{公開URL}
#ダーツ #マイセッティング #dartsspec
```

- ハッシュタグは `#ダーツ #マイセッティング #dartsspec` の 3 つで固定する。
- セッティング名を含めることで、タイムライン上でも投稿ごとの個性が出るようにする。

## 6. 動的 OGP（最重要の仕掛け）

> SNS で URL が貼られた際に、そのセッティング内容が **1 枚のスタイリッシュな画像**として表示される。CGM マーケティングにおいて最も重要な仕掛けである。

### 6.1 要件

- 公開 URL（`/s/{id}`）を X・Discord・Slack 等に貼ると、セッティング内容が反映された **1200×630px の OGP 画像**が表示される。
- 画像はセッティングごとに**動的生成**される（静的な共通画像ではない）。
- 生成内容: セッティング名・作者・総重量・全長・各パーツ名を、セッティングカードと同じトーンでレイアウト。

### 6.2 実装方針

- **Cloudflare Workers 上でのエッジ動的生成**を用いる。[`workers-og`](https://github.com/kvnang/workers-og)（Satori→SVG→resvg→PNG を WASM で完結・`@vercel/og` 風 API）を採用する。HTML/CSS ライクな記述から PNG を生成できる。
- OGP 画像エンドポイント: TanStack Start の server route `GET /api/og/{setting_id}` → PNG を返す（[04 §5](./04-architecture.md)）。
- 公開ページ（`s/$id`）のルートで、`loader` が取得したセッティングから以下のメタタグを出力する。

```html
<meta property="og:title"       content="〇〇スペシャル | darts spec" />
<meta property="og:description" content="総重量 20.5g / 全長 65.0mm" />
<meta property="og:image"       content="https://dartsspec.example/api/og/{id}" />
<meta property="og:url"         content="https://dartsspec.example/s/{id}" />
<meta name="twitter:card"       content="summary_large_image" />
```

### 6.3 OGP レイアウト（1200×630・確定）

以下を正式レイアウトとして**確定**する（#16）。Satori の制約（Flexbox のみ・グリッド不可・`position: absolute` は限定的）を前提に、**縦方向の Flex 一列**で構成する。

```
┌──────────────────────────────────────────────┐
│  🎯 darts spec                                 │  ← ヘッダ行
│                                                │
│   〇〇スペシャル              by @user          │  ← タイトル行
│                                                │
│   20.5 g  /  65.0 mm     ← 巨大に表示           │  ← 数値行
│                                                │
│   BARREL  TRiNiDAD Gomez Type 12               │
│   SHAFT   L-style 190 Silent                   │  ← パーツ 4 行
│   FLIGHT  Fit Flight Standard                  │
│   TIP     COSMO Fit Point PLUS                 │
└──────────────────────────────────────────────┘
```

| 要素 | 確定仕様 |
|---|---|
| キャンバス | 1200×630、背景 `--bg-base`、外周 padding 64px、左上から右下へ `--bg-base`→`--bg-surface` の微グラデーション可 |
| ヘッダ行 | 「🎯 darts spec」ロゴテキスト（絵文字はテキストとして描画）。24px・`--accent` |
| タイトル行 | セッティング名 44px・`--text-primary`・最大 1 行（溢れは省略記号）。右端に「by @user」20px・`--text-secondary` |
| 数値行 | `{重量} g / {全長} mm` を 96px・weight 700・tabular figures・`--text-primary`。単位は 40px・`--text-secondary` |
| パーツ 4 行 | カテゴリラベル（20px・大文字・`--accent`・幅固定 140px）+ ブランド `--text-secondary` + 製品名 `--text-primary`（26px）。未選択は「—」 |
| フォント | Noto Sans JP（サブセット埋め込み）。数値も同フォントの太ウェイトで代用可（フォント 2 本を避けバンドルを節約） |

- 画面表示用セッティングカード（§4.2）と同じトークン・構成比率を用い、「カードがそのまま画像になった」印象を保つ。

### 6.4 OGP 実装上の注意

- **キャッシュ**: 同一セッティングの OGP は変更が無い限りキャッシュする（`Cache-Control` / Cloudflare Cache）。OGP は SNS から高頻度に叩かれるが強くキャッシュされるため、実質「キャッシュウィンドウごとに 1 回」の生成で済む。編集時はキャッシュを破棄する（URL に更新版クエリを付与、または再検証）。
- **フォント**: 動的画像生成では Web フォントを明示的に読み込む必要がある。日本語フォントは容量が大きいため、必要字形のサブセット化を検討する（Worker バンドルサイズにも効く）。
- **フォールバック**: セッティングが非公開 / 削除済みの場合は、汎用のサービス OGP を返す。
- **X のキャッシュ**: X は OGP を強くキャッシュする。開発中は Card Validator 相当での確認と、URL 更新でのキャッシュ回避を想定しておく。

#### Cloudflare Workers 特有の落とし穴（workers-og / Satori + resvg-wasm）

事前に把握しておくべき既知の罠でござる（[04 §1](./04-architecture.md) のエッジ留意点と対応）。

- **WASM は動的コンパイル不可**: Satori/resvg の WASM は static import で wrangler に事前コンパイルさせる。動的 `compile` は Workers で失敗する。
- **画像の自動 fetch が無言で失敗**: Satori は画像 URL を自前で fetch しようとすると Workers 上で静かに失敗する。**画像は手動で fetch → base64 data URL に変換**して埋め込む。
- **PNG / JPEG のみ**: WebP を渡すと不親切なエラーで落ちる。ロゴ等の埋め込み画像は PNG/JPEG に統一する。

## 7. アクセシビリティ / レスポンシブ

- スマホファースト。シェアからの流入は大半がモバイル。
- ダークテーマでもコントラスト比を確保（本文テキストは WCAG AA 目安）。
- タップ領域は最低 44×44px。セレクトボックスはモバイルで扱いやすいネイティブ or カスタム UI を選定。

## 8. デザイン成果物の管理

- カラー・タイポ・コンポーネントは、実装時に **Tailwind の theme トークン**または CSS 変数として一元管理する。
- デザインカンプが必要な場合は Figma 等で作成し、本ドキュメントからリンクする（未作成なら本ドキュメントが暫定仕様）。
