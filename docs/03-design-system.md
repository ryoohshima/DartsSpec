# 03. デザインシステム / UI・UX

対応ステップ: **Step 3 — デザインシステムと UI/UX の設計**

既存のダーツ関連サイト（ガチャガチャした EC サイト風）にはない「モダンさ」が最大の武器になる。**ここの設計品質がサービスの差別化の core** であるため、徹底して作り込む。

## 1. ビジュアルの方向性

**ダークモードを基調とした、サイバーで洗練されたデザイン**を第一候補とする。金属バレルの質感・ダーツバーの雰囲気と親和性が高く、数値やスペックが映える。

### デザイン原則

1. **Less is More** — 情報を詰め込まず、余白でスペックを際立たせる。
2. **数値が主役** — 総重量・全長といった数字を、大きく・美しく見せる（プレイヤーが最も気にする情報）。
3. **静的でなく動的** — パーツ選択に対して数値がアニメーションで反応し、「触って楽しい」を演出。
4. **シェア前提** — あらゆる画面が「スクショしたくなる / 貼りたくなる」美しさを持つ。

## 2. カラートークン（ダークテーマ・初期案）

具体的な値は実装時に微調整するが、方向性を固定するための初期パレット。

| トークン | 用途 | 参考値 |
|---|---|---|
| `--bg-base` | 背景（最暗） | `#0A0A0F` |
| `--bg-surface` | カード面 | `#14141C` |
| `--bg-elevated` | 浮いた要素 | `#1E1E2A` |
| `--text-primary` | 主要テキスト | `#F5F5F7` |
| `--text-secondary` | 補助テキスト | `#9A9AAF` |
| `--border` | 境界線 | `#2A2A38` |
| `--accent` | アクセント（CTA・強調） | `#00E5C7`（サイバー系ティール） |
| `--accent-alt` | 第二アクセント | `#7B5CFF`（バイオレット） |
| `--danger` | エラー | `#FF5C7A` |

> アクセントは「ネオン」に寄せると安っぽくなりやすい。彩度を抑えた**上品なネオン**を狙う。ライトテーマは初期スコープ外とし、まずダーク一本に集中する。

## 3. タイポグラフィ

| 用途 | 方針 |
|---|---|
| 数値（総重量・全長） | 太めのサンセリフ / 等幅寄り。桁が揃うフォントで「計器」感を出す |
| 見出し | モダンなサンセリフ（例: Inter / Noto Sans JP） |
| 本文 | 可読性重視の日本語対応フォント（Noto Sans JP 等） |

- 数字は **tabular figures（等幅数字）** を使い、アニメーション時にガタつかないようにする。

## 4. 主要画面と UI コンポーネント

### 4.1 画面一覧

| 画面 | 役割 | 認証 |
|---|---|---|
| ランディング（LP） | サービス紹介・登録導線 | 不要 |
| サインアップ / ログイン | 認証 | 不要 |
| セッティング作成 / 編集 | パーツ選択・リアルタイム合算 | 必要 |
| マイページ（一覧） | 自分のセッティング一覧 | 必要 |
| 公開セッティングページ | シェア先が見る画面・OGP 対象 | 不要（閲覧） |
| 作成完了 / シェア画面 | シェアボタンを大きく配置 | 必要 |

### 4.2 中核コンポーネント: セッティングカード

本サービスの顔となる「カード型 UI」。テキストと数値ベースで、1 枚で 1 セッティングを美しく表現する。

```
┌─────────────────────────────────────┐
│  MY SETTING                          │
│  「〇〇スペシャル」        by @user   │
│                                      │
│   TOTAL WEIGHT      TOTAL LENGTH     │
│     20.5 g            65.0 mm        │  ← 数値が主役・大きく
│                                      │
│  ─────────────────────────────────  │
│  BARREL   TRiNiDAD  Gomez Type 12    │
│  SHAFT    L-style   190 Silent       │
│  FLIGHT   Fit Flight Standard        │
│  TIP      COSMO     Fit Point PLUS   │
│                                      │
│              [ X でシェア ]           │
└─────────────────────────────────────┘
```

- パーツ行はアイコン + ブランド + 製品名で構成。
- 総重量・全長は最上部に大きく。プレイヤーが一番見たい情報を最優先配置。

### 4.3 インタラクティブ合算表示

- パーツをセレクトすると、総重量・全長の数字が**カウントアップ / カウントダウンのアニメーション**で変動する。
- 実装は数値補間（例: framer-motion の `useSpring` / `animate` や `react-countup` 相当）。
- 「触って数字が動く」体験そのものが、シェアの動機になる。

## 5. 「シェアしたくなる」動線設計

CGM マーケティングの生命線。シェアの摩擦を極限まで下げる。

- **マイページ・作成完了画面**に、「X（旧 Twitter）にマイセッティングを晒す」文脈のシェアボタンを**大きく**配置する。
- シェアボタンは Web Intent（`https://twitter.com/intent/tweet?...`）で、あらかじめ気の利いた定型文 + ハッシュタグ（例: `#ダーツ #マイセッティング`）+ 公開 URL を差し込む。
- コピー用の「URL をコピー」ボタンも併設する。

### 定型文の例

```
自分のダーツセッティングを晒してみた🎯
総重量 20.5g / 全長 65.0mm

{公開URL}
#ダーツ #マイセッティング #DartsSpec
```

## 6. 動的 OGP（最重要の仕掛け）

> SNS で URL が貼られた際に、そのセッティング内容が **1 枚のスタイリッシュな画像**として表示される。CGM マーケティングにおいて最も重要な仕掛けである。

### 6.1 要件

- 公開 URL（`/s/{id}`）を X・Discord・Slack 等に貼ると、セッティング内容が反映された **1200×630px の OGP 画像**が表示される。
- 画像はセッティングごとに**動的生成**される（静的な共通画像ではない）。
- 生成内容: セッティング名・作者・総重量・全長・各パーツ名を、セッティングカードと同じトーンでレイアウト。

### 6.2 実装方針

- **エッジでの動的画像生成**を用いる。Next.js なら `next/og`（`ImageResponse`）や `@vercel/og` が第一候補。HTML/CSS ライクな記述から PNG を生成できる。
- OGP 画像エンドポイント例: `GET /api/og/{setting_id}` → PNG を返す。
- 公開ページの `<head>` に以下のメタタグを設定する。

```html
<meta property="og:title"       content="〇〇スペシャル | DartsSpec" />
<meta property="og:description" content="総重量 20.5g / 全長 65.0mm" />
<meta property="og:image"       content="https://dartsspec.app/api/og/{id}" />
<meta property="og:url"         content="https://dartsspec.app/s/{id}" />
<meta name="twitter:card"       content="summary_large_image" />
```

### 6.3 OGP レイアウト（1200×630）

```
┌──────────────────────────────────────────────┐
│  🎯 DartsSpec                                  │
│                                                │
│   〇〇スペシャル              by @user          │
│                                                │
│   20.5 g  /  65.0 mm     ← 巨大に表示           │
│                                                │
│   BARREL  TRiNiDAD Gomez Type 12               │
│   SHAFT   L-style 190 Silent                   │
│   FLIGHT  Fit Flight Standard                  │
│   TIP     COSMO Fit Point PLUS                 │
└──────────────────────────────────────────────┘
```

### 6.4 OGP 実装上の注意

- **キャッシュ**: 同一セッティングの OGP は変更が無い限りキャッシュする（`Cache-Control` / エッジキャッシュ）。編集時はキャッシュを破棄する（URL に更新版クエリを付与、または再検証）。
- **フォント**: 動的画像生成では Web フォントを明示的に読み込む必要がある。日本語フォントは容量が大きいため、必要字形のサブセット化を検討する。
- **フォールバック**: セッティングが非公開 / 削除済みの場合は、汎用のサービス OGP を返す。
- **X のキャッシュ**: X は OGP を強くキャッシュする。開発中は [Card Validator](https://cards-dev.twitter.com/validator) 相当での確認と、URL 更新でのキャッシュ回避を想定しておく。

## 7. アクセシビリティ / レスポンシブ

- スマホファースト。シェアからの流入は大半がモバイル。
- ダークテーマでもコントラスト比を確保（本文テキストは WCAG AA 目安）。
- タップ領域は最低 44×44px。セレクトボックスはモバイルで扱いやすいネイティブ or カスタム UI を選定。

## 8. デザイン成果物の管理

- カラー・タイポ・コンポーネントは、実装時に **Tailwind の theme トークン**または CSS 変数として一元管理する。
- デザインカンプが必要な場合は Figma 等で作成し、本ドキュメントからリンクする（未作成なら本ドキュメントが暫定仕様）。
