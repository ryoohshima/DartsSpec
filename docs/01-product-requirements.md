# 01. プロダクト要求仕様（PRD）

対応ステップ: **Step 1 — 企画の言語化とミニマム要件の決定（仕様の引き算）**

個人開発において最速でリリースするため、「何ができて、何をやらないか」の境界線を引く。

## 1. コア価値

> 自分のダーツセッティング（バレル・シャフト・フライト・チップ）を記録でき、**URL ひとつで SNS に美しくシェアできる**。

この一点に機能を絞る。あれもこれもと詰め込まず、「記録」と「美しい共有」の 2 つだけを磨き込む。

## 2. ターゲットユーザー

| ペルソナ | 状況 | 求めるもの |
|---|---|---|
| 熱量の高いアマチュア | 自分のセッティングを人に見せたい・記録したい | 綺麗にまとまる・シェアが映える |
| プロ / インフルエンサー | ファンに使用機材を共有したい | 信頼できる正確なスペック・ブランド網羅 |
| 初〜中級者 | 上手い人のセッティングを参考にしたい | 他人の公開ページを見て真似できる |

初期の主戦場は **X（旧 Twitter）のダーツコミュニティ** である。

## 3. ダーツセッティングの構成要素

ユーザーが記録する「1 セッティング」は、以下のパーツで構成される。

| パーツ | 役割 | 主なスペック |
|---|---|---|
| **バレル (Barrel)** | 手で握る金属部分。セッティングの主役 | 重量(g)・全長(mm)・最大径(mm)・素材 |
| **シャフト (Shaft)** | バレルとフライトをつなぐ棒 | 長さ(mm)・素材 |
| **フライト (Flight)** | 羽根。飛行を安定させる | 形状(シェイプ)・厚み |
| **チップ (Tip / Point)** | 先端。ソフト（樹脂）/ ハード（金属） | 種類・長さ(mm) |

> 補足: 厳密には「チップ」はソフトダーツの樹脂先端、「ポイント」はハードダーツの金属先端を指す。MVP では区分を持たせつつ、UI 上は「チップ/ポイント」としてまとめて扱う。

## 4. MVP スコープ（必ずやること）

### 4.1 機能要件

- [ ] **ユーザー登録 / ログイン機能**
  - メール + パスワード、および OAuth（Google）でのサインイン。X（旧 Twitter）OAuth は初期リリース対象外・将来検討。
- [ ] **パーツのセレクトボックス選択機能**
  - バレル・シャフト・フライト・チップを、マスタデータからプルダウンで選択。
- [ ] **スペックの自動合算ロジック**
  - 選択されたパーツから **総重量**・**セッティング全長** を自動計算し、リアルタイム表示。
- [ ] **マイページの保存**
  - 作成したセッティングをユーザーに紐づけて永続化。1 ユーザーが複数セッティングを保持可能。
- [ ] **固有の公開 URL（シェア用）の発行**
  - 例: `https://dartsspec.app/s/{setting_id}`。ログイン不要で誰でも閲覧可能。
- [ ] **動的 OGP**
  - 公開 URL を SNS に貼ると、セッティング内容が 1 枚のスタイリッシュな画像として表示される。（詳細は [03](./03-design-system.md)）

### 4.2 非機能要件（MVP 最低ライン）

| 項目 | 目標 |
|---|---|
| レスポンス | 主要ページの初期表示 2 秒以内 |
| モバイル対応 | スマホファースト（閲覧・共有はスマホが主） |
| 可用性 | 個人開発レベル（無料〜低額ホスティングで可） |
| セキュリティ | 認証情報のハッシュ化・HTTPS・公開範囲の適切な制御 |

## 5. スコープ外（初期リリースではやらないこと）

意図的に「やらない」と決めることで、リリースを最速化する。

- **3D / 高度なグラフィックによるビジュアルシミュレーション**
  - 初期はテキストと数値ベースのモダンな「カード型 UI」で十分成立する。
- **パーツへのレビュー機能・コメント機能**
  - CGM の中心はあくまで「セッティングの共有」。相互コミュニケーションは後回し。
- **EC サイトのような購入導線の構築**
  - アフィリエイトや購入リンクは初期には組み込まない。
- **セッティングの検索 / ランキング / タイムライン**
  - まずは「作る・シェアする」の一本道。発見体験は将来フェーズ。

> これらは Issue 化して将来バックログに積む。判断に迷ったら「やらない」を選ぶ。

## 6. ユーザーストーリー（MVP）

```
1. プレイヤーはサービスに登録・ログインする。
2. 「新規セッティング作成」からバレル/シャフト/フライト/チップを選ぶ。
3. 選ぶそばに総重量・全長がアニメーション付きで更新される。
4. セッティング名を付けて保存する。
5. 完了画面で「X にマイセッティングを晒す」ボタンを押す。
6. X に貼られた URL が、スタイリッシュなカード画像（OGP）で表示される。
7. リンクを見た別のプレイヤーが公開ページを閲覧し、自分も作ってみたくなる。
```

## 7. 成功指標（初期 KPI 案）

| 指標 | 意味 |
|---|---|
| セッティング作成数 | コア機能が使われているか |
| 公開 URL のシェア数 / クリック数 | CGM 拡散が回っているか |
| 登録ユーザー数 | 母数の成長 |
| 「パーツが無くて離脱」率 | データ網羅性の不足検知（[02](./02-parts-data.md) と連動） |

## 8. 未確定事項（要意思決定）

- サービス正式名称・ドメイン（本ドキュメントでは仮に `DartsSpec` / `dartsspec.app`）。
- 非ログインユーザーがセッティング作成を「お試し」できるか（保存時にログイン誘導する UX の可否）。

> 決定済み: 初期リリースの認証方式は **メール+パスワード + OAuth（Google のみ）**。X（旧 Twitter）OAuth は将来検討とする（詳細は [04-architecture.md §4](./04-architecture.md)）。
