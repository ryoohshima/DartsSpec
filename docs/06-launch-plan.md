# 06. ローンチ計画

対応ステップ: **Step 5 — テストとコミュニティへのローンチ**

作ったものをいきなり広告で大々的に宣伝せず、**熱量の高いコミュニティに直接届ける**形で段階的に広げる。CGM は「最初の良質な投稿」が呼び水になるため、初速の質を重視する。

## 1. フェーズ全体像

```
Phase 1  クローズドテスト   → 身近な仲間・一部インフルエンサーで検証・改善
Phase 2  初期ローンチ       → X のダーツコミュニティへストーリー投稿
Phase 3  コンテンツ蓄積     → プロ/熱量アマに個別アプローチし良質投稿を溜める
Phase 4  グロース           → 口コミ・OGP 拡散のループを回す
```

## 2. Phase 1: クローズドテスト

- **対象**: 身近なダーツ仲間、SNS で影響力のある一部のプレイヤー。
- **目的**: バグ・使いにくい点の洗い出しと修正。
- **やること**:
  - [ ] 5〜15 名程度に先行して触ってもらう。
  - [ ] 「自分のパーツが無い」報告を集め、[02. パーツデータ](./02-parts-data.md) のマスタを追加。
  - [ ] 作成〜シェアの導線でつまずく箇所を観察し、UX を修正。
  - [ ] OGP が各人の環境（X / LINE / Discord）で正しく出るか確認。
- **完了条件**: 主要導線に致命的バグがなく、「パーツが無い」離脱リスクが十分下がっている。

### フィードバック収集の観点

| 観点 | 見るポイント |
|---|---|
| データ網羅 | 使っているパーツが見つかるか |
| 分かりやすさ | 説明なしで作成〜シェアまで辿れるか |
| 美しさ | 「シェアしたい」と思えるか（OGP 含む） |
| 速度 | ストレスない表示速度か |

## 3. Phase 2: 初期ローンチ（SNS 中心）

- **主戦場**: X（旧 Twitter）のダーツコミュニティ。
- **語り口**: 開発者自身のストーリーで届ける。

> 「既存サイトが古くて不便だったので、自分用にモダンなツールを作りました」

- **やること**:
  - [ ] 開発者自身のマイセッティングを、動的 OGP 付きで投稿（プロダクトの実物をそのまま見せる）。
  - [ ] 開発の動機・こだわり（モダンさ・シェアの手軽さ）をスレッドで語る。
  - [ ] ハッシュタグ（`#ダーツ #マイセッティング` 等）を活用。
  - [ ] プロフィール / 固定ポストにサービス URL を掲示。

### ローンチ投稿の型（例）

```
既存のダーツサイトが古くて不便だったので、
自分のセッティングをURLひとつで綺麗にシェアできるツールを作りました🎯

総重量も全長も自動計算、SNSに貼るとこんな感じのカードになります👇

{自分の公開URL（OGPカードが展開される）}
#ダーツ #マイセッティング
```

## 4. Phase 3: コンテンツ蓄積（CGM の初速づくり）

CGM プラットフォームは「先に良質な投稿が並んでいる」ことが新規ユーザーの信頼と動機になる。

- **やること**:
  - [ ] プロプレイヤー・熱量あるアマチュアに**個別にアプローチ**。
    > 「セッティングをこのツールで綺麗にまとめてみませんか？」
  - [ ] 協力者のセッティングをプラットフォーム内に蓄積し、初期の良質コンテンツを揃える。
  - [ ] 反応の良い投稿を観察し、シェアされやすい要素（見た目・文言）を UI/OGP に反映。

## 5. Phase 4: グロースのループ

CGM の拡散は「投稿 → OGP がタイムラインで映える → 見た人が作る → また投稿」の循環で回る。

```
   ┌───────────────────────────────────────┐
   │                                       │
   ▼                                       │
ユーザーが作成 ─▶ 公開URLをXにシェア ─▶ OGPカードが映える
   ▲                                       │
   │                                       ▼
   └──── 見た人が「自分も」と登録 ◀── リンクをクリックして閲覧
```

- このループの各接点（作成のしやすさ / OGP の美しさ / 閲覧から登録への誘導）を継続的に磨くことが、広告に頼らないグロースの本質である。

## 6. ローンチ後に検討する拡張（スコープ外だったもの）

初期リリースで意図的に外した機能を、需要を見ながらバックログから引き上げる（[01. PRD §5](./01-product-requirements.md)）。

| 機能 | 引き上げの判断材料 |
|---|---|
| セッティング検索 / ランキング | 投稿数が増え「他人の設定を探したい」需要が出たら |
| フォロー / タイムライン | リピート利用と回遊が増えたら |
| レビュー / コメント | コミュニティの交流ニーズが確認できたら |
| EC / アフィリエイト導線 | 十分なトラフィックが集まり収益化の段階で |

## 7. KPI と観測

[01. PRD §7](./01-product-requirements.md) の指標を継続観測する。

- セッティング作成数 / 公開 URL のシェア・クリック数 / 登録ユーザー数。
- 「パーツが無くて離脱」の兆候（作成中断・検索空振り）を拾い、マスタ拡充にフィードバック。
- OGP のクリック率は、シェアループが機能しているかの最重要シグナル。

## 8. ローンチ準備チェックリスト

- [ ] 本番環境が安定稼働（[05. 開発計画](./05-development-plan.md) のリリース前チェック完了）
- [ ] 主要 5 ブランドのパーツが十分に揃っている
- [ ] 動的 OGP が X / Discord / LINE で正しく表示される
- [ ] 開発者自身のサンプルセッティングが数件公開済み
- [ ] ローンチ投稿の文面・スレッドを準備済み
- [ ] 問い合わせ / バグ報告の受け口（X DM / Issue 等）を用意
